# Release Process

This project uses GitHub Releases as the changelog surface and PyPI publishing
trigger. Releases must be tag-first so `release-notes-scribe` can draft the
GitHub Release notes before the release is published.

## Required Local Checks

Run the standard validation loop before tagging:

```bash
pdm run check
```

`pdm run check` runs:

```bash
pdm run pytest
pdm run ruff check src tests
pdm build
```

## Process

1. Choose the next semantic version tag, for example `v0.3.2`.
2. Create and push the tag:

```bash
git tag v0.3.2
git push origin v0.3.2
```

3. Wait for the `Draft Release Notes` GitHub Actions workflow to complete.
4. Review the drafted GitHub Release generated by
   `btfranklin/release-notes-scribe`.
5. Publish the GitHub Release.
6. Confirm the `Upload Python Package` workflow completes.

## Ordering Constraint

Do not publish the GitHub Release before pushing the tag and letting the
`Draft Release Notes` workflow run. Release notes are generated by the
tag-triggered workflow in `.github/workflows/draft-release-notes.yml`.

Publishing the GitHub Release triggers `.github/workflows/python-publish.yml`,
which builds with PDM and publishes to PyPI through trusted publishing. The
publishing workflow does not generate release notes.
